> 阶段之间可以重构代码，但尽量避免。前期以跑通基础功能为主。

## 阶段 1：直接调用 LLM（不用任何框架）

**新概念（2个）：**
- OpenAI API 的调用方式（发消息、收回复）
- Prompt 是什么，怎么写

**实现内容：**
- `POST /chat` — 接收用户消息，调用 DeepSeek/OpenAI，返回回复
- 支持 `stream=True` 流式响应（像打字机一样逐字输出）
- 消息历史保存到数据库

**为什么先做这个：**
在用任何框架之前，先亲手感受"原始 LLM 调用"是什么感觉。
后续引入框架时，才能理解框架帮你省掉了什么。

**阶段结束后你能回答：**
- LLM 的输入输出是什么格式？
- `system` / `user` / `assistant` 消息分别是什么意思？
- 流式输出和普通输出有什么区别？

## 阶段 2：引入 LangGraph — 最简单的线性图

**新概念（3个）：**
- 什么是"图"（Graph）：节点 + 边
- LangGraph 的 State（全局状态对象）
- 节点（Node）是一个普通 Python 函数

**实现内容：**
- 把阶段 1 的 `/chat` 改造成一个只有两个节点的图：
  `接收输入` → `调用LLM` → `保存结果`
- 不做任何分支，只是线性流动

**为什么这样设计：**
LangGraph 最难的地方是"状态"概念。
先用最简单的线性图建立直觉，不要一上来就搞分支。

**阶段结束后你能回答：**
- LangGraph 的图和普通函数调用有什么区别？
- State 在节点之间是怎么传递的？

## 阶段 3：工具调用（Tool Use）

**新概念（2个）：**
- Function Calling：LLM 可以"决定"调用外部工具
- 工具的输入/输出如何接入图

**实现内容：**
- 接入 Tavily Search 作为第一个工具
- LLM 自己判断：这个问题需要搜索吗？
- 图结构：
  ```
  调用LLM → （需要用工具？）→ 执行搜索 → 再次调用LLM → 保存
                ↓否
              直接保存
  ```

**为什么这样设计：**
工具调用是 Agent 和普通聊天的本质区别。
这一步完成后，你的程序才算真正意义上的 "Agent"。

**阶段结束后你能回答：**
- LLM 怎么知道什么时候该用工具？
- 工具的返回结果怎么传回给 LLM？

## 阶段 4：意图识别 + 路由

**新概念（2个）：**
- 条件边（Conditional Edge）：根据条件走不同分支
- 意图分类（用 LLM 判断用户想做什么）

**实现内容：**
- 新增 `classify_intent` 节点，识别用户意图：
  - `chat`（普通问答）
  - `create_plan`（制定学习计划）
  - `quiz`（出题练习）
- 根据意图路由到不同处理节点（后两个先返回占位回复）
- 意图写入数据库

**为什么这样设计：**
条件路由是 Agent 工作流的灵魂。
学完这步，你才算真正理解"Agent 是怎么做决策的"。

**阶段结束后你能回答：**
- 条件边和普通边有什么区别？
- 意图识别失败了怎么处理？

## 阶段 5：结构化输出 — 学习计划生成

**新概念（2个）：**
- `with_structured_output`：让 LLM 直接输出符合 Pydantic 格式的 JSON
- 结构化数据的持久化

**实现内容：**
- 实现 `create_plan` 意图的完整处理：
  - 用 Pydantic 定义 `LearningPlan` 数据结构（阶段、知识点、时长）
  - LLM 输出结构化 JSON，不再是自由文本
  - 计划存入数据库
- `GET /plan/{session_id}` — 查看生成的学习计划

**为什么这样设计：**
让 LLM 输出"可被程序处理的数据"是工程落地的关键。
自由文本只能给人看，结构化输出才能驱动后续逻辑。

**阶段结束后你能回答：**
- 结构化输出和普通文本输出的原理区别是什么？
- LLM 输出不符合格式时怎么处理？

## 阶段 6：记忆 — 跨会话上下文

**新概念（2个）：**
- LangGraph Checkpointer（图状态快照持久化）
- 会话记忆的加载与恢复

**实现内容：**
- 用 SQLite 存储 LangGraph 的 checkpoint（每次对话后保存图状态）
- 用户下次对话时自动加载历史上下文，无需重新介绍自己
- `GET /history/{user_id}` — 查看学习历史

**为什么这样设计：**
有了记忆才是真正的"私教"。
这是 Agent 区别于普通聊天机器人的核心特性之一。

**阶段结束后你能回答：**
- Checkpointer 保存的是什么？
- 多用户场景下，状态是怎么隔离的？

## 阶段 7：RAG — 知识库检索

**新概念（3个）：**
- Embedding（把文本变成向量）
- 向量数据库 Chroma（按语义相似度检索）
- RAG 模式：检索 → 注入 Context → 生成回答

**实现内容：**
- 用 Chroma 存储用户上传的学习资料
- 回答问题前先检索相关片段，再交给 LLM
- `POST /knowledge` — 上传学习资料（纯文本）
- 检索结果作为 context 注入 prompt，并在回答中附上来源

**为什么这样设计：**
RAG 是目前企业 LLM 落地最常见的方案。
学完这步，你能解决 LLM 幻觉和知识时效性两个核心问题。

**阶段结束后你能回答：**
- 为什么不直接把所有资料都塞进 prompt？
- 相似度检索和关键词检索有什么区别？

## 阶段 8：反思节点（高级，视情况）

**新概念（1个）：**
- 反思循环（Reflection Loop）：让 LLM 检查并改进自己的输出

**实现内容：**
- 在图中加一个 `reflect` 节点
- LLM 检查自己的回答：是否完整？是否有幻觉？
- 不满足质量要求则重新生成，满足则输出
